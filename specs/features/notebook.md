# 帳簿（Notebook）管理功能規格

## 1. Overview

本功能涵蓋 Splitwise 風格 PWA 中的**共用帳簿**：建立與維護帳簿中繼資料、以 **8 字元邀請代碼** 邀請／加入成員、成員角色（擁有者／成員）、離開或移除成員，以及**帳簿列表**與**帳簿詳情**兩個核心頁面的資訊架構與呈現。資料存取透過 Supabase Client 與 RLS，表結構與查詢合約分別以 `../db-schema.md`、`../api-contracts.md` 為單一真相來源。

**範圍摘要**

- 建立帳簿：`name`、`description`、`default_currency`（ISO 4217）、`icon`  
- 擁有者可編輯／刪除帳簿  
- 邀請：分享 `invite_code`（8 字元隨機碼）  
- 加入：輸入邀請代碼加入為 `member`  
- 角色：`owner`、`member`  
- 成員離開；擁有者可移除成員  
- 列表頁：成員數、幣別、餘額摘要  
- 詳情頁：支出依**日期**分組顯示

## 2. User stories

| 編號 | 身為… | 我想要… | 以便… |
|------|--------|---------|--------|
| N-01 | 已登入使用者 | 建立新帳簿並設定名稱、描述、預設幣別與圖示 | 與朋友分帳記帳 |
| N-02 | 帳簿擁有者 | 編輯帳簿基本資料 | 更正名稱或預設幣別等 |
| N-03 | 帳簿擁有者 | 刪除帳簿 | 在不需要時清除整本帳（含關聯資料） |
| N-04 | 擁有者 | 取得並分享邀請代碼 | 邀請他人加入 |
| N-05 | 已登入使用者 | 輸入邀請代碼加入帳簿 | 參與既有帳簿 |
| N-06 | 成員 | 查看自己在帳簿中的角色 | 知道權限範圍 |
| N-07 | 成員（非擁有者） | 離開帳簿 | 不再參與分帳 |
| N-08 | 擁有者 | 移除某位成員 | 管理成員名單 |
| N-09 | 使用者 | 在列表頁看到每本帳簿的摘要 | 快速選擇要進入的帳簿 |
| N-10 | 使用者 | 在詳情頁依日期瀏覽支出 | 對照實際消費時程 |

## 3. Data model（參考 db-schema.md）

以下實體與約束以 **`../db-schema.md`** 為準，實作與 RLS 不得與其矛盾。

| 實體 | 章節 | 與本功能相關重點 |
|------|------|------------------|
| `profiles` | §2.1 | `id`、`display_name`、`avatar_url`；成員列表顯示用 |
| `notebooks` | §2.2 | `name`、`description`、`default_currency`（ISO 4217）、`icon`、`invite_code`（UNIQUE、8 字元）、`created_by` |
| `notebook_members` | §2.3 | `notebook_id`、`user_id`、`role`（`owner` \| `member`）、`UNIQUE (notebook_id, user_id)` |
| `expenses` | §2.4 | 詳情頁分組依 `expense_date`；列表頁「餘額」計算需與分攤／結算邏輯一致（見 §6） |
| `expense_splits` | §2.5 | 分攤明細；淨額／餘額可由此與 `expenses` 衍生 |
| `settlements` | §2.6 | 若有「已結清」調整，餘額應反映結算（產品定義見 §6） |

**擁有者唯一性**：每本帳簿應至少有一位 `owner`；建議由資料庫 trigger 在建立帳簿時自動插入 `notebook_members`（`role = 'owner'`），與 `../api-contracts.md` §4.2 描述一致。若擁有者離開或刪除帳簿的規則有特殊限制，見 §7。

| 行為 | 參考章節 |

## 6. Business logic

### 6.1 建立帳簿

- 必填：`name`、`default_currency`（有效 ISO 4217 代碼，可由前端下拉或 autocomplete 限制）。  
- 選填：`description`、`icon`（若未傳，使用 `../db-schema.md` 預設，例如 `'book'`）。  
- **`invite_code`**：由前端產生 **8 字元**隨機碼（字元集需與後端 UNIQUE 碰撞處理策略一致：插入失敗則重試產生）。  
- `created_by` 設為目前使用者 `profiles.id`。  
- 建立成功後，依合約 trigger 插入 `notebook_members`（`role = 'owner'`）。

### 6.2 編輯與刪除（僅擁有者）

- **編輯**：可更新 `name`、`description`、`default_currency`、`icon`；不建議在一般 UI 直接改 `invite_code`（若需「重設代碼」為進階功能，需防舊連結失效並提示）。  
- **刪除**：僅 `owner` 可執行；刪除 `notebooks` 列後，依 schema CASCADE 清除成員、支出、分攤、結算。  
- RLS 應保證非擁有者無法 `update`／`delete` 帳簿本體。

### 6.3 邀請與加入

- **邀請**：擁有者（或產品允許的成員）展示目前 `invite_code` 供複製；本質為分享秘密代碼，非 Email 邀請信（除非另開功能）。  
- **加入**：使用者輸入代碼 → `eq('invite_code', code)` 查單一帳簿 → 若存在且尚未在 `notebook_members`，則 `insert` `role = 'member'`。  
- 若已為成員，應顯示「你已在此帳簿」並可導向詳情。

### 6.4 角色與成員管理

- **owner**：一人或多人由產品決定；本 schema 允許多位 owner 時，刪除權限需與 RLS 一致。若僅允許單一 owner，應在 DB 或應用層保證。  
- **離開**：使用者刪除自己的 `notebook_members` 列。  
- **移除成員**：僅擁有者可刪除他人之 `notebook_members` 列；**不可**移除最後一位擁有者而不處理轉移（見 §7）。  
- 擁有者**離開**：需先轉移擁有權或刪除帳簿（推薦產品規則，見 §7）。

### 6.5 列表頁摘要：成員數、幣別、餘額

- **成員數**：`notebook_members` 該 `notebook_id` 之列數。  
- **幣別**：顯示 `notebooks.default_currency`（若淨額計算使用多幣別，需轉換或分幣別顯示—**MVP** 可先假設列表淨額以 `default_currency` 呈現，與 PRD 一致時再擴充）。  
- **餘額（淨額）**：對目前使用者，在該帳簿內基於 `expense_splits` 與 `expenses.paid_by` 計算「我代付與我應分攤之差」，並扣除／加上已記錄之 `settlements`（若產品定義結算沖銷分攤）。**精確公式**應與 `../PRD.md` 或財務規格一致；本文件要求列表與詳情使用同一套計算，避免數字不一致。

### 6.6 詳情頁：支出依日期分組

- 資料來源：`../api-contracts.md` §5.1，依 `expense_date` 降序（或升序，團隊定一種後全 App 一致）。  
- UI：同一日期的多筆支出放在同一標題下；標題格式本地化。  
- 每筆卡片顯示：描述、付款人、總額、目前使用者與該筆相關的「欠／應收」摘要（與分攤邏輯一致）。

## 7. Edge cases

- **邀請代碼碰撞**：插入帳簿時 UNIQUE 衝突 → 重新產生代碼重試（上限次數與錯誤提示）。  
- **無效或過期代碼**：查無帳簿 → 顯示「找不到帳簿」。  
- **已加入重複加入**：唯一約束違反 → 友善提示，不建立重複列。  
- **最後一位擁有者離開或移除自己**：應禁止，或強制先轉移 `owner`／刪除帳簿。  
- **移除成員時該成員仍有未結清分攤**：產品可選擇禁止移除、或允許移除但保留歷史支出紀錄（由 RLS 決定是否仍「看得到」）；需在 PRD 明確。  
- **刪除帳簿**：需二次確認；說明將刪除所有支出與成員關係。  
- **幣別變更**：若帳簿已有大量不同 `currency` 的支出，變更 `default_currency` 僅影響預設新支出與列表顯示策略，不應自動改寫歷史列。  
- **RLS 拒絕**：顯示權限不足，而非空白頁。

## 8. Acceptance criteria

1. 已登入使用者可建立帳簿，包含 `name`、`description`（選填）、`default_currency`（ISO 4217）、`icon`，且新帳簿具唯一 `invite_code`（8 字元），建立者成為 `owner`（與 `../db-schema.md`、`../api-contracts.md` §4.2 一致）。  
2. 擁有者可更新帳簿中允許的欄位；非擁有者無法從 UI 成功提交更新（RLS 亦拒絕）。  
3. 擁有者可刪除帳簿，且相關 `notebook_members`、`expenses` 等依 CASCADE 清除。  
4. 使用者可透過正確的 `invite_code` 加入帳簿為 `member`；錯誤代碼有明確錯誤狀態。  
5. 成員可離開帳簿；擁有者可移除其他成員（在 §7 規則允許範圍內）。  
6. 帳簿列表頁每列顯示成員數、`default_currency`（或合約之顯示幣別），以及對目前使用者的淨餘額摘要，且與詳情頁計算邏輯一致。  
7. 帳簿詳情頁以**日期**分組列出支出，分組內順序與 API 排序一致，單筆卡片資訊可辨識付款人與分攤摘要。  
8. 邀請流程可讓擁有者複製 8 字元代碼；加入流程可從列表進入並完成加入。  
9. 所有讀寫操作符合 `../api-contracts.md` 所列方法與參數；資料表與約束符合 `../db-schema.md`。
