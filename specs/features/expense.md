# 支出管理功能規格

## 1. Overview

本文件描述類 Splitwise 之 **支出新增、分帳、列表與編修刪除** 行為，適用於帳簿內成員共同記帳的 PWA。資料持久化使用 **Supabase (PostgreSQL + RLS)**，與專案共用之 schema 與 API 合約一致。

**範圍摘要**：

- 新增支出：金額、幣別、描述、分類、付款人、消費日期、備註，以及分帳方式與分攤明細。
- 分帳類型：**均分**、**自訂比例**、**自訂金額**。
- 分類：餐飲、交通、住宿、購物、娛樂、其他（對應 DB 列舉值）。
- 編輯／刪除：僅 **支出建立者** 或 **帳簿 owner**（與 RLS 一致）。
- 列表：**依消費日期遞減**，並 **依日期分組** 顯示。

## 3. Data model（參考 `../db-schema.md`）

以下欄位與表格以 `../db-schema.md` 為準，實作不得與 schema CHECK／FK 衝突。

### 3.1 `expenses`

| 概念 | 欄位 | 說明 |
|------|------|------|
| 金額 | `amount` | `numeric(12,2)`，必須大於 0 |
| 幣別 | `currency` | ISO 4217，預設與帳簿或產品預設一致 |
| 描述 | `description` | 必填 |
| 分類 | `category` | `food` \| `transport` \| `lodging` \| `shopping` \| `entertainment` \| `other` |
| 分帳方式 | `split_type` | `equal` \| `percentage` \| `exact` |
| 付款人 | `paid_by` | FK → `profiles.id` |
| 消費日期 | `expense_date` | `date`，預設當日 |
| 備註 | `note` | 可為空 |

**建立者與權限**：`../db-schema.md` 第 5.4 節 RLS 規定 **UPDATE／DELETE** 僅限 **支出建立者** 或 **帳簿 owner**。若目前資料表未持久化建立者識別，實作須與 schema／RLS 同步（例如新增 `created_by` 或由審計機制還原），本功能之前端按鈕顯示與錯誤處理應與後端規則一致。

### 3.2 `expense_splits`

每筆支出對應多筆分攤（每位分攤人至多一列，`UNIQUE (expense_id, user_id)`）。

| 欄位 | 說明 |
|------|------|
| `user_id` | 分攤人 |
| `amount` | 該成員在此支出中的分攤金額（與支出 **相同幣別**），≥ 0 |
| `is_settled` | 結清狀態（本文件不展開結算流程） |

**分攤成員集合**：預設為該帳簿當下成員中、使用者勾選要參與分帳的人員（至少一人）。均分／比例／自訂金額皆只產生 **被勾選成員** 的 `expense_splits` 列。

## 4. API contracts（參考 `../api-contracts.md`）

資料讀寫透過 Supabase Client，合約見 **`../api-contracts.md`**：

- **列表**：§ 5.1 取得帳簿支出（含 `paid_by`、`expense_splits`），排序以 **消費日期** 為主；前端再依日期分組。
- **新增**：§ 5.2 先 `insert` `expenses`，再 `insert` `expense_splits`（金額由本文件第 6 節計算）。
- **更新**：§ 5.3 更新 `expenses` 後，對該 `expense_id` **先刪除** 既有 splits 再 **批次插入** 新 splits。
- **刪除**：§ 5.4 刪除 `expenses`（CASCADE 刪除 splits）。

Realtime 可選：§ 7.1 訂閱支出變更，列表需能反映他人新增／修改。

**錯誤處理**：若 RLS 拒絕 UPDATE／DELETE，應向使用者顯示無權限，而非靜默失敗。

## 6. Business logic（公式與演算法）

以下金額運算均以支出幣別 **同一小數刻度** 進行（與 `numeric(12,2)` 一致：**最小單位 0.01**）。記號：`T` = 支出總額（大於 0），`n` = 參與分帳人數（至少 1 人）。

### 6.1 均分（`equal`）

目標：將 `T` 分為 `n` 份，每份為整分單位，且 **n 份加總等於 T**。

建議演算法（最小貨幣單位整數運算）：

1. 令 `cents = round(T * 100)`（或依幣別使用專用刻度，本專案非 JPY/KRW 支出列仍存 `numeric(12,2)`；若未來支援零小數幣別作為支出幣別，需改為以該幣別最小單位整數運算）。
2. `base = floor(cents / n)`，`remainder = cents - base * n`。
3. 將參與者以 **穩定順序** 排列（例如 `user_id` 字串排序），前 `remainder` 人分得 `base + 1`，其餘分得 `base`。
4. 寫入 `expense_splits.amount = 分得金額 / 100`。

**性質**：每人分額相差至多 0.01；總和嚴格等於 `T`。

### 6.2 自訂比例（`percentage`）

使用者為每位參與者輸入 **非負** 比例 `p_i`（單位：百分比數值，例如 40 代表 40%）。

**驗證**：`sum(p_i) = 100`（容許浮點誤差時，建議以 **整數萬分比** 或 **有理數** 校驗，例如 `abs(sum - 100) < 1e-6` 或改為儲存整數比例和為 10000）。

**分攤金額**（先以分為單位）：

1. `cents = round(T * 100)`。
2. 對每人：`raw_i = cents * p_i / 100`，`alloc_i = floor(raw_i)`，記錄 `frac_i = raw_i - alloc_i`。
3. `sum_alloc = Σ alloc_i`，`diff = cents - sum_alloc`（0 ≤ diff &lt; n 類似）。
4. 將 `diff` 個 **0.01** 依 `frac_i` **由大到小** 分配給參與者（平手時依穩定順序），使最終 `Σ amount_i = T`。

若產品要求比例僅為整數百分比，可限制輸入為整數並仍用上述餘額分配。

### 6.3 自訂金額（`exact`）

使用者直接輸入每位參與者的分攤金額 `a_i`（與支出同幣別）。

**驗證**：

- 各 `a_i ≥ 0`。
- `Σ a_i = T`（與 DB 總額一致，容許 **至多 ±0.01** 的捨入誤差時，應以前端提示調整或強制校準最後一人；建議 **強制相等** 以避免帳不平）。

### 6.4 與 `paid_by` 的關係

分攤僅描述「每人應承擔額」；誰先墊付由 `paid_by` 表示。結欠矩陣／結算不在本文件計算，但 **splits 總和必須等於 `expenses.amount`**。

## 7. Edge cases

| 情境 | 行為 |
|------|------|
| 僅一人參與分帳 | 允許；該人分攤 = `T`（均分／比例 100%／自訂全額） |
| 均分時 `n` 無法整除 | 使用 6.1 餘數分配，避免四捨五入導致總和 ≠ T |
| 比例總和 ≠ 100% | 阻擋送出並提示修正 |
| 自訂金額總和 ≠ T | 阻擋送出；標示差額 |
| 參與者為 0 | 不允許送出 |
| 無權編輯／刪除 | 隱藏或 disabled 按鈕；若 API 回傳 RLS 錯誤，顯示權限說明 |
| 編輯時成員已離開帳簿 | 已存在之 `expense_splits` 仍保留歷史 `user_id`；新支出僅能選目前成員（產品可另定「幽靈成員」顯示策略） |
| 並發編輯 | 最後寫入優先；可搭配 Realtime 提示列表過期 |
| `amount` 或 split 輸入非數字／負數 | 前端擋下；後端 CHECK 再防 |

## 8. Acceptance criteria

- [ ] 新增支出可填寫：**金額、幣別、描述、分類（六類之一）、墊付人、消費日期、備註**，並成功寫入 `expenses`。
- [ ] 分帳可選 **均分／自訂比例／自訂金額**，且產生之 `expense_splits` 筆數等於勾選之參與人數。
- [ ] **均分** 時，所有分攤額加總 **嚴格等於** 支出總額，且每人 ≥ 0。
- [ ] **自訂比例** 時，比例總和為 100%（依產品定義之容差），分攤加總等於總額。
- [ ] **自訂金額** 時，各項 ≥ 0 且加總等於總額。
- [ ] 列表資料 **依 `expense_date` 遞減**，UI **依日期分組** 顯示。
- [ ] **建立者** 與 **帳簿 owner** 可編輯／刪除；其他人不可（與 RLS 一致）。
- [ ] 編輯支出後，splits 與後端合約一致：**先刪後建**，且與新分帳邏輯一致。
- [ ] 分類顯示名稱：food→餐飲、transport→交通、lodging→住宿、shopping→購物、entertainment→娛樂、other→其他。
